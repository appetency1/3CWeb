# Systematic Debugging

> 遇到任何 Bug、测试失败或意外行为时使用。**核心原则：先找根因再修复。**
> 来源：Trae skills / `systematic-debugging`

## 铁律

```
没有根因调查，就不能提议修复。
```

## 四阶段流程

### Phase 1: 根因调查
1. **收集信息** — 错误信息、复现步骤、环境
2. **缩小范围** — 二分法定位问题区域
3. **阅读相关代码** — 涉及路径的完整逻辑链
4. **提出假设** — 可能的根因解释
5. **验证假设** — 用数据/log/测试确认
6. **确认根因** — 精准定位到 file:line

### Phase 2: 修复方案设计
- 最小改动原则
- 评估影响范围
- 考虑副作用

### Phase 3: 实施修复
- 执行修改
- 验证根因场景已解决
- 确认无回归

### Phase 4: 预防
- 添加测试覆盖
- 记录学到的东西

## 常见反模式
- ❌ 看到症状就猜修复
- ❌ 连续尝试多个修复而不验证
- ❌ 修复后不验证原始场景
- ❌ 修复表面问题而非根因
